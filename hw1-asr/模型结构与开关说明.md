# GLM-ASR 模型结构与 FUSED / BACKEND 开关说明

---

## 模型整体结构

```
音频输入 (mel spectrogram)
       │
       ▼
┌─────────────────────────────────────────┐
│  Audio Encoder  (32 层 Transformer)     │
│                                         │
│  每层：                                 │
│    LayerNorm                            │
│    Self-Attention                       │
│      └─ q_proj / k_proj / v_proj       │  ← Linear
│      └─ o_proj                          │  ← Linear
│    LayerNorm                            │
│    EncoderMLP                           │
│      └─ fc1  (hidden→intermediate)      │  ← Linear + GELU
│      └─ fc2  (intermediate→hidden)      │  ← Linear
│                                         │
│  hidden=1280, heads=20, ff=5120         │
└─────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────┐
│  Projector  (4-frame 池化 + 2层 Linear) │
│                                         │
│    Linear(5120 → 4096)  + GELU          │  ← Linear
│    Linear(4096 → 3584)                  │  ← Linear
└─────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────┐
│  Text Decoder  (28 层 Transformer)      │
│                                         │
│  每层：                                 │
│    RMSNorm                              │
│    GQA Self-Attention (28Q / 4KV)       │
│      └─ q_proj / k_proj / v_proj       │  ← Linear
│      └─ o_proj                          │  ← Linear
│    RMSNorm                              │
│    MLP (SwiGLU)                         │
│      └─ gate_proj (hidden→inter)        │  ← Linear + SiLU ┐
│      └─ up_proj   (hidden→inter)        │  ← Linear        ├─ 合并
│      └─ down_proj (inter→hidden)        │  ← Linear        │
│                                         │
│  hidden=3584, heads=28Q/4KV, ff=18944  │
└─────────────────────────────────────────┘
       │
       ▼
    文字输出
```

## `EncoderMLP.FUSED` 在动哪里

**作用范围：Audio Encoder 32层，每层的 `fc1` + GELU**

```
EncoderMLP._forward_standard（FUSED=False）:
  x → fc1(Linear) → [写回HBM] → GELU → [写回HBM] → fc2(Linear)

EncoderMLP._forward_fused（FUSED=True）:
  x → linear_gelu_kernel → [直接store] → fc2(Linear)
       └─ fc1 matmul + GELU 在 register 里一次完成，不写中间结果
```

**不影响：** fc2、q/k/v/o_proj、Projector、Decoder 任何层

---

## `MLP.FUSED` 在动哪里

**作用范围：Text Decoder 28层，每层的 `gate_proj` + `up_proj` + SiLU + ×**

```
MLP._forward_standard（FUSED=False）:
  x → gate_proj(Linear) → [写回HBM]
  x → up_proj(Linear)   → [写回HBM]
  SiLU(gate) * up        → [写回HBM]
  → down_proj(Linear)

MLP._forward_fused（FUSED=True）:
  x → swiglu_fused_kernel → [直接store intermediate]
       └─ gate matmul + up matmul + SiLU + × 全在 register 里完成
  → down_proj(Linear)   ← 这一步永远走 Linear.__call__，不受FUSED影响
```

**不影响：** down_proj、q/k/v/o_proj、Encoder、Projector

---

## `Linear.BACKEND` 在动哪里

**作用范围：所有 `Linear.__call__` 的调用**（FUSED 接管的那些 Linear 除外）

| BACKEND 值 | 实际走法 |
|---|---|
| `"cublas"` / `"torch"` | `x_2d @ self.weight.t()`（PyTorch 调 cuBLAS） |
| `"triton"` | `linear_kernel_tf32`（手写 Triton tiled matmul） |

**受影响的层（FUSED=True 时）：**

| 位置 | 受影响的 Linear |
|---|---|
| Audio Encoder ×32 | q/k/v/o_proj, **fc2** |
| Projector | Linear(5120→4096), Linear(4096→3584) |
| Text Decoder ×28 | q/k/v/o_proj, **down_proj** |

> 注意：fc1（被 EncoderMLP.FUSED 接管）和 gate_proj/up_proj（被 MLP.FUSED 接管）**不走** `Linear.__call__`，所以 BACKEND 对它们无效。

---

## 四种组合总结

| Config | gate/up/fc1 | 其余 Linear（fc2/down/proj/attn） |
|---|---|---|
| cublas + FUSED=off | cuBLAS ×2 + 分离激活 | cuBLAS |
| cublas + FUSED=on  | Triton fused kernel   | cuBLAS |
| triton + FUSED=off | cuBLAS ×2 + 分离激活 | **Triton** matmul |
| triton + FUSED=on  | Triton fused kernel   | **Triton** matmul |

实验中 Config B = `cublas + FUSED=on`，Config T1 = `triton + FUSED=on`。

## 快速记忆口诀

- **FUSED** = 把 MLP 里 "多步激活操作" 塞进一个 kernel，减少 HBM 读写
- **BACKEND=triton** = 把"除了attention剩下所有矩阵乘法"从 cuBLAS 换成手写 Triton matmul,attention 里面的 矩阵乘法 默认 用 tl.dot
- **Flash** = 把 attention 中间计算（Q@K^T + softmax + @V）换成 FlashAttention kernel
- 三者完全正交，互不重叠，可以独立开关

## Attention 的完整流程与三个开关的关系

```
x
├─ q_proj(x)  → Linear.__call__ → 由 BACKEND 控制（cuBLAS 或 Triton matmul）
├─ k_proj(x)  → Linear.__call__ → 由 BACKEND 控制
└─ v_proj(x)  → Linear.__call__ → 由 BACKEND 控制
       │
       ▼
【Flash=off】naive attention（三个独立 Triton kernel）:
  attention_scores_kernel: scores = tl.sum(k * q, axis=1)  → 写完整 seq×seq 矩阵到 HBM
  softmax_inplace_kernel:  softmax(scores)                  → 读+写 HBM
  attention_output_kernel: output = tl.sum(v * w, axis=0)  → 读 HBM

【Flash=on】FlashAttention（一个 Triton kernel）:
  flash_attention_kernel:
    for tile in seq_k:
      load K tile, V tile（一小块，进 SRAM）
      tl.dot(Q, K_tile) → scores（留在 SRAM）
      online softmax    → 留在 SRAM，不写 HBM
      tl.dot(P, V_tile) → 累加到 acc（留在 SRAM）
    store 最终 output → 只写一次 HBM
       │
       ▼
o_proj(·)  → Linear.__call__ → 由 BACKEND 控制
```

**关键：BACKEND 只管 q/k/v/o_proj 这四个 Linear，attention 中间的矩阵乘法完全独立，不走 BACKEND。**

## Naive vs FlashAttention 的 Q@K^T 实现差异

| | Q@K^T 实现 | IO 复杂度 |
|---|---|---|
| Naive | `tl.sum(k * q, axis=1)`，逐行点积，每次处理一个 query | O(seq²) 写回 HBM |
| Flash | `tl.dot(Q_tile, K_tile)`，tiled matmul，BLOCK_M × BLOCK_N | O(seq) 写回 HBM |

Naive 用的是逐元素点积（不是 `tl.dot()`），Flash 才是真正的 tiled matmul（用 `tl.dot()`）。

**为什么 naive 不用 `tl.dot()` 而是逐行点积？**

不是有意为之，而是硬件约束驼逼的：`tl.dot()` 要求 tile 的两个维度均 ≥ 16（tensor core 约束）。decode 阶段 seq_q=1，Q 只有 1 行，无法满足这个要求。逐行点积（`tl.sum(k * q)`）对 seq_q=1 和 seq_q=很大都能工作，是更通用的写法。

FlashAttention 通过强制 BLOCK_M ≥ 16（decode 阶段会 padding 成 16 行再 mask 掉多余部分）绕开了这个限制，因此可以用 `tl.dot()`。

**naive 的"tiling"和 linear 的 tiling 有何不同？**

- `linear_kernel_tf32`：按 (BLOCK_M × BLOCK_N) 的 output tile 分配 thread block，每个 tile 沿 K 维 loop 累加，是标准 2D tiled GEMM
- `attention_scores_kernel`：每个 thread block 只处理 **一个 query 位置**，沿 seq_k 方向扫描所有 key，是向量×矩阵（GEMV）风格，不是 GEMM

两者结构完全不同，naive attention 更接近 GEMV，而 FlashAttention 才是真正的 tiled GEMM。

## 为什么 FUSED=on 在 cublas 下反而更慢

`swiglu_fused_kernel` 内部自己写了 tiled matmul loop（用 `tl.dot()`），**不调用 cuBLAS**：

```
cublas + FUSED=off:
  gate_proj → cuBLAS matmul（极快）→ 写 HBM
  up_proj   → cuBLAS matmul（极快）→ 写 HBM
  silu + ×  → 额外 kernel

cublas + FUSED=on:
  swiglu_fused_kernel → 内嵌 Triton tl.dot()（较慢）→ 直接 store，省 HBM roundtrip
```

FUSED 的收益（省 HBM）< 损失的 compute（cuBLAS → 手写 Triton），整体变慢。

---

## 为什么 FUSED=on 在 triton 下预期比 FUSED=off 更快（但仍慢于 cublas+FUSED=off）

```
triton + FUSED=off:
  linear_kernel_tf32(gate)  → Triton tl.dot()，写 HBM
  linear_kernel_tf32(up)    → Triton tl.dot()，写 HBM
  silu_kernel               → 额外 kernel

triton + FUSED=on：
  swiglu_fused_kernel → 内嵌同等质量的 Triton tl.dot()，省 HBM roundtrip
```

matmul 质量持平（都是手写 Triton），FUSED 额外省 HBM，净收益为正。

## tl.dot vs cuBLAS vs FlashAttention 的本质区别

| | compute 速度 | IO 节省 | 适用场景 |
|---|---|---|---|
| cuBLAS (`nn.Linear`) | ★★★★★ 极快 | 无 | 大矩阵乘法 |
| 手写 Triton (`tl.dot` + tiling) | ★★★ 较慢 | 无（除非 FUSED） | 可控制 tiling，可 fuse 激活 |
| FlashAttention (`tl.dot` + online softmax) | ★★★ 较慢（compute） | ★★★★★ 极大（O(seq²)→O(seq)） | attention，memory-bound 场景 |

**核心教训：**
- Triton 的意义**不是**超越 cuBLAS 的 matmul compute 速度
- Triton 的意义是实现 cuBLAS **结构性做不到**的操作（如 FlashAttention 的 online softmax + fused matmul）
- Attention 是 memory-bound 操作，IO 节省的量级远超 compute 差距
- F3（FlashAttention）比 A（naive）快 43%，而 T1（Triton matmul）比 A 只慢 1.5%——IO 收益远大于 compute 损失
